Auto Memory: Taming the 200-Line MEMORY.md Limit

2026-03-20
memory auto-memory advanced
Quick Quiz
What happens when your MEMORY.md exceeds 200 lines, and how should you structure it to avoid losing context?
Click to reveal answer
Claude Code only loads the first 200 lines of MEMORY.md — everything beyond that is silently truncated. To avoid losing context, keep MEMORY.md as a concise index of short entries and pointers, and move detailed content into separate topic files (like debugging.md or api-conventions.md) in the same memory directory.

What Is Auto Memory?

Auto memory is a 2026 addition to Claude Code that lets Claude autonomously save learnings across sessions. When Claude discovers something useful — a build command, an architectural pattern, a correction you gave it — it writes that knowledge to MEMORY.md without you having to say "remember this."

It's on by default. The file lives at:

~/.claude/projects/<encoded-path>/memory/MEMORY.md

Every subsequent session in that project loads this file automatically, giving Claude a running knowledge base about your project.

The 200-Line Hard Limit

Here's the catch: Claude Code only injects the first 200 lines of MEMORY.md into context. Everything past line 200 is truncated silently. If your most important memories are buried at the bottom, Claude will never see them.

This limit exists because every line of MEMORY.md consumes tokens from your context window on every single conversation turn. A bloated memory file means less room for actual code reasoning.

The Topic File Strategy

The solution is to treat MEMORY.md as an index, not a dumping ground. Keep it lean with short entries and pointers, then offload detailed content into separate topic files:

~/.claude/projects/<encoded-path>/memory/
  MEMORY.md              # Index — stays under 200 lines
  debugging.md           # Deep debugging notes
  api-conventions.md     # API patterns and endpoints
  architecture.md        # System design decisions

Your MEMORY.md becomes a table of contents:

# MEMORY.md

## Build & Dev
- `pnpm dev` starts all services
- Port 5432 conflicts with local Postgres — use 5433

## Architecture
- Monorepo managed by NX
- See architecture.md for service boundaries

## Gotchas
- See debugging.md for common build failures

The Capture-Promote-Prune Cycle

Auto memory is additive — Claude appends but never reorganizes. Left unchecked, MEMORY.md becomes a disorganized list. Use this three-phase cycle:

  • Capture: Let auto memory and /memory collect raw learnings during sessions
  • Promote: Weekly, move validated facts into your project's CLAUDE.md (team-shared, version-controlled)
  • Prune: Monthly, delete stale or redundant entries from MEMORY.md

The goal: MEMORY.md stays a lean scratch pad. Stable knowledge graduates to CLAUDE.md. Stale knowledge gets deleted.

Key Takeaway

Keep MEMORY.md under 200 lines by using it as an index with pointers to topic files — and regularly promote validated learnings to CLAUDE.md so your best knowledge is always team-shared and permanent.
