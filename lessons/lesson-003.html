Hooks: Auto-Run Scripts Before and After Every Tool Call

2026-03-23
hooks settings automation
Quick Quiz
What are the two hook types in Claude Code, and where do you configure them?
Click to reveal answer
preTool (runs before a tool executes) and postTool (runs after). Configure them in the "hooks" key of your settings file: .claude/settings.json for project-level or ~/.claude/settings.json for global.

What Are Hooks?

Hooks are shell commands that Claude Code runs automatically before or after specific tool invocations. They let you enforce policies, auto-format code, log activity, or block dangerous operations — without relying on Claude to remember to do it.

Think of them like git hooks, but for Claude Code's tools (Bash, Edit, Read, Write, etc.).

The Two Hook Types

  • preTool — fires before a tool executes. Use it for validation, audit logging, or blocking operations.
  • postTool — fires after a tool completes. Use it for auto-formatting, notifications, or cleanup.

How to Configure

Add a "hooks" key to your settings JSON. Each hook specifies which tool it targets and the command to run:

{
  "hooks": {
    "preTool": [
      {
        "tool": "Bash",
        "command": "echo \"$(date) Bash tool called\" >> /tmp/claude-audit.log"
      }
    ],
    "postTool": [
      {
        "tool": "Edit",
        "command": "npx eslint --fix $CLAUDE_FILE_PATH"
      }
    ]
  }
}

This config does two things: logs every Bash invocation to an audit file, and auto-runs ESLint on any file Claude edits.

Where to Put It

  • Project-level: .claude/settings.json — shared with your team via git
  • Project-local: .claude/settings.local.json — personal overrides, not committed
  • Global: ~/.claude/settings.json — applies to all projects

Practical Examples

Auto-Format After Every Edit

{
  "hooks": {
    "postTool": [
      {
        "tool": "Edit",
        "command": "npx prettier --write $CLAUDE_FILE_PATH"
      }
    ]
  }
}

Block Edits to Sensitive Files

# In your preTool command:
if echo "$CLAUDE_FILE_PATH" | grep -q '.env'; then
  echo "BLOCKED: Cannot modify .env files" >&2
  exit 1
fi

A non-zero exit code from a preTool hook blocks the tool from executing — acting as a policy gate.

Things to Watch Out For

  • Hooks run synchronously and add latency to every matching tool call. Keep them fast.
  • A failing preTool hook blocks the tool — useful for gates, but make sure you don't accidentally block legitimate work.
  • Debug hook issues with the --debug hooks flag to see execution details.
  • For complex logic, write an external script and call it from the hook command.

Key Takeaway

Use preTool hooks to gate and audit, postTool hooks to auto-format and clean up — they enforce your standards automatically so Claude doesn't have to remember them.
