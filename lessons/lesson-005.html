Worktrees: Run Parallel Claude Sessions Without Conflicts

2026-03-25
worktrees parallel-sessions workflow
Quick Quiz
How do you run multiple Claude Code sessions on the same repo without them stepping on each other's files?
Click to reveal answer
Use the --worktree flag (claude --worktree my-feature). Each session gets its own isolated copy of the repo where it can edit files, run builds, and create commits without affecting your main working directory or other sessions.

The Problem: One Repo, Multiple Tasks

You want Claude to refactor the auth module and write new API tests at the same time. But if both sessions share the same working directory, they'll overwrite each other's changes, conflict on builds, and generally make a mess.

The Fix: --worktree

Git worktrees let you check out the same repo into multiple directories simultaneously, sharing the same .git object store. Claude Code has first-class support for this via the --worktree flag:

# Named worktree — creates an isolated checkout
claude --worktree auth-refactor

# Auto-named worktree
claude --worktree

# Worktree + tmux session (great for monitoring)
claude --worktree api-tests --tmux

Each worktree is a full, independent checkout. Builds, tests, and linters in one worktree don't interfere with another.

Practical Parallel Workflow

Here's how to run three tasks simultaneously:

# Terminal 1: Refactor auth
claude --worktree auth-refactor -p "Refactor auth to use JWT tokens"

# Terminal 2: Write tests
claude --worktree test-coverage -p "Add missing tests for src/utils/"

# Terminal 3: Fix CI
claude --worktree ci-fix -p "Fix the failing lint step in CI"

Each session works in its own directory, creates commits on its own branch, and won't conflict with the others until you're ready to merge.

Script It for Batch Work

For repetitive parallel tasks, script the pattern:

#!/bin/bash
tasks=(
  "Review src/api/ for security issues"
  "Add TypeScript types to src/models/"
  "Generate tests for src/utils/"
)

for i in "${!tasks[@]}"; do
  claude --worktree "task-$i" \
    -p "${tasks[$i]}" \
    --max-budget-usd 1.00 \
    > "results/task-$i.md" 2>&1 &
done
wait
echo "All tasks complete"

Cleanup

Worktrees persist after sessions end so you can review the changes. When you're done:

# List worktrees
git worktree list

# Remove a finished worktree
git worktree remove auth-refactor

Custom Branch Naming

By default, --worktree auto-generates branch names. If your team enforces a convention like user/JIRA-123-feature, you have two options.

Option 1: Create the Worktree Yourself

Use standard Git to create the worktree with your exact naming, then launch Claude inside it:

# Create a worktree with your branch naming convention
git worktree add ../feature-auth -b jacques/JIRA-123-feature-auth

# Launch Claude inside it
cd ../feature-auth && claude

This gives you full control over the branch name, directory location, and which base branch to fork from. Best when you only need it occasionally.

Option 2: Use a WorktreeCreate Hook

If you want to keep using claude --worktree directly while enforcing your naming convention every time, hook into the worktree lifecycle. Claude passes a JSON payload via stdin with the name and cwd, so your script can construct the branch name however you need.

// .claude/settings.json
{
  "hooks": {
    "WorktreeCreate": [
      {
        "command": ".claude/hooks/worktree-name.sh",
        "timeout": 10000
      }
    ]
  }
}

Your hook script receives the worktree name and can rewrite it:

#!/bin/bash
# .claude/hooks/worktree-name.sh
# Reads JSON payload from stdin: { "name": "...", "cwd": "..." }

PAYLOAD=$(cat)
NAME=$(echo "$PAYLOAD" | jq -r '.name')
USER=$(git config user.name | tr ' ' '-' | tr '[:upper:]' '[:lower:]')

# Construct branch name with your convention
echo "{\"branch\": \"${USER}/${NAME}\"}"

Now every claude --worktree auth-refactor automatically creates a branch like jacques/auth-refactor without any extra steps.

Which Approach to Pick

Approach Setup Best For
Manual worktree None One-off tasks, choosing a specific base branch
WorktreeCreate hook Once Team-wide conventions enforced on every --worktree

When to Use Worktrees vs. Subagents

  • Worktrees — for tasks that need independent builds, tests, or long-running sessions. Each gets a full checkout.
  • Subagents (the Agent tool inside a session) — for quick, focused subtasks like research or single-file edits that share the parent's working directory.

Key Takeaway

Use claude --worktree <name> to run parallel Claude sessions in isolated repo checkouts — no file conflicts, no build interference, full independence.
