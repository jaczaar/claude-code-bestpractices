Custom Slash Commands: Build Reusable Prompt Templates

2026-03-27
custom-commands productivity advanced
Quick Quiz
How do you create a custom slash command that accepts arguments, and where does it live?
Click to reveal answer
Create a markdown file in .claude/commands/ (project) or ~/.claude/commands/ (user). Use the $ARGUMENTS placeholder in the template to capture everything typed after the command name. Invoke it with /project:name args or /user:name args.

What Are Custom Slash Commands?

Custom slash commands are markdown files that act as reusable prompt templates. When you type /project:test, Claude loads the contents of .claude/commands/test.md and uses it as your prompt. The filename (minus .md) becomes the command name.

Two Scopes: Project vs User

  • Project commands live in .claude/commands/ and are invoked as /project:<name>. Commit them to git so your whole team benefits.
  • User commands live in ~/.claude/commands/ and are invoked as /user:<name>. These are personal and work across all repos.
  • Nested commands use subdirectories: .claude/commands/review/security.md becomes /project:review:security.

The $ARGUMENTS Placeholder

The real power comes from $ARGUMENTS. It captures everything typed after the command name, making your templates flexible.

Example — create .claude/commands/test.md:

Write unit tests for: $ARGUMENTS

Requirements:
- Use the existing test framework in this project
- Cover happy path and edge cases
- Include error cases
- Follow existing test patterns in the codebase

Now run:

/project:test the authentication middleware

Claude receives the full template with $ARGUMENTS replaced by "the authentication middleware".

Practical Examples

A standup summary command (~/.claude/commands/standup.md):

Summarize my git activity from the last 24 hours:
1. List commits with messages
2. List files changed
3. Highlight any open PRs

Format as bullet points for a standup update.

A targeted code explainer (.claude/commands/explain.md):

Explain $ARGUMENTS in plain English:
- What it does
- Why it exists (infer from context)
- Key dependencies
- Potential gotchas

Keep it under 200 words.

Best Practices

  • One command, one task. Keep templates focused — don't build mega-commands.
  • Keep templates concise. Every word consumes context tokens when invoked.
  • Include constraints. Be explicit about output format, coding standards, and stop conditions.
  • Use / + letters to filter the command list in real time — custom commands appear alongside built-in ones.

Key Takeaway

Custom slash commands turn your best prompts into one-keystroke workflows — drop a markdown file in .claude/commands/, use $ARGUMENTS for flexibility, and commit them to git so your whole team benefits.
