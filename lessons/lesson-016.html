Quick Quiz
What does /simplify do, and how does it differ from /review?
Click to reveal answer
/simplify reviews your changed code for reuse, quality, and efficiency — then automatically fixes the issues it finds. Unlike /review, which only gives feedback, /simplify edits your code directly. It never changes behavior — only how the code achieves it.

The Last Step Before You Commit

You've just finished a feature. The tests pass. But you know there's cleanup to do — duplicated logic, verbose patterns, opportunities to reuse existing utilities. You could review it yourself, or you could let Claude do it in seconds.

The /simplify skill, introduced by Claude Code creator Boris Cherny alongside /batch (@bcherny on X), is a built-in skill that reviews your changed code for reuse opportunities, code quality issues, and efficiency improvements — then applies the fixes automatically.

How It Works

Just type /simplify after making changes:

/simplify

Claude scans your changed files (the diff, not the whole codebase), looking for three things:

  • Reuse: Are you re-implementing something that already exists in your codebase? Claude finds the existing utility and swaps it in.
  • Quality: Overly complex conditionals, deeply nested logic, unclear naming — Claude restructures for clarity.
  • Efficiency: Redundant computations, unnecessary allocations, or patterns that could be expressed more concisely.

The key guarantee: /simplify never changes what your code does. If your function returns a specific value, handles certain edge cases, or produces particular side effects, all of that stays exactly the same. Only the how changes.

/simplify vs. /review

These two skills are complementary, not interchangeable:

  • /review gives you feedback — comments, suggestions, potential issues. You decide what to act on.
  • /simplify takes action — it edits your code directly. Review the diff, accept or reject.

A solid workflow: run /simplify first to clean up the obvious stuff, then /review for a deeper architectural pass.

Supercharge It with CLAUDE.md

Add a project-specific review checklist to your CLAUDE.md and /simplify will respect it:

## Code review checklist
- Prefer early returns over nested if/else
- Use our shared logger (src/utils/logger.ts), never console.log
- Destructure props in function signatures
- Max function length: 40 lines

Now /simplify enforces your team's conventions automatically — not just generic best practices.

Where /simplify Fits in Your Workflow

  • Before committing: Run /simplify as your personal code cleanup pass
  • Inside /batch: Each parallel agent runs /simplify automatically before creating its PR (as covered in Lesson 015)
  • After prototyping: Write messy code fast to explore an idea, then /simplify to clean it up

Key Takeaway

Run /simplify before every commit to automatically clean up reuse, quality, and efficiency issues in your changed code — pair it with a review checklist in CLAUDE.md to enforce your team's specific conventions.
