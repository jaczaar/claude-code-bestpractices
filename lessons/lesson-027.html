Agent Teams: Orchestrate a Swarm of Claude Sessions on One Task

2026-05-06
agent-teams workflow multi-agent
Quick Quiz
How do Agent Teams differ from subagents, and when should you use them?
Click to reveal answer
Subagents run inside one session and can only report back to the parent. Agent Teams spawn separate Claude Code instances with their own context windows, a shared task list with dependency tracking, and a mailbox for direct inter-agent messaging. Use them when your task has clearly separable domains (frontend, backend, tests) that benefit from parallel, coordinated work.

What Are Agent Teams?

Agent Teams — also called "swarm mode" — let you coordinate multiple Claude Code sessions on a single task. One session acts as the lead, breaking the work into subtasks and assigning them to teammates. Each teammate gets its own context window, can read and write files, run commands, and talk directly to other teammates.

As shared by @bcherny on X, Agent Teams shipped as an experimental feature alongside the Opus 4.6 release in February 2026.

How to Enable It

Agent Teams are experimental and off by default. Enable them in one of two ways:

# Option 1: Environment variable
CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 claude

# Option 2: Add to your .claude/settings.json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}

Requires Claude Code v2.1.32 or later.

How It Works

Three primitives make teams different from plain subagents:

  • Shared task list with dependencies. The lead creates tasks that can block each other. When a blocking task completes, downstream tasks automatically unblock. Teammates self-claim the next available task when they finish.
  • Mailbox messaging. The lead can message any teammate, teammates can message the lead, and teammates can message each other directly. This two-way communication is what separates teams from fire-and-forget subagents.
  • Independent context windows. Each teammate has its own full context, so a frontend agent and a backend agent never compete for the same token budget.

When to Use Teams

Agent Teams burn more tokens than a single session. Use them when the payoff justifies the cost:

  • Cross-layer features — frontend, backend, and test changes each owned by a different teammate.
  • Research and review — multiple teammates investigate different aspects of a problem, then share and challenge findings.
  • Debugging with competing hypotheses — teammates test different theories in parallel and converge on the answer.

For routine single-domain tasks, a regular session (or worktrees with separate sessions) is more cost-effective.

Best Practice: Clean Domain Separation

Teams work best when you can cleanly separate domains. Give each teammate a clear scope — one handles src/api/, another handles src/components/, a third writes tests. Overlap leads to merge conflicts and wasted tokens. Think of it like assigning tickets to engineers: clear ownership, minimal overlap.

Key Takeaway

Agent Teams turn Claude Code into a coordinated squad — use them for cross-layer features and parallel research where clean domain separation makes the extra tokens worthwhile.
