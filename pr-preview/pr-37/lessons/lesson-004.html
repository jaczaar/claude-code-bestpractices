Lesson 004

Tame Context Bloat with /compact Focus Hints

2026-03-25
context-management workflow productivity
Quick Quiz
Why should you run /compact with a focus hint before auto-compression triggers, rather than letting Claude Code compress automatically?
Click to reveal answer
Auto-compression uses generic summarization with no guidance on what matters. A manual /compact with a focus hint (e.g., /compact focus on the API contract changes) tells Claude exactly which details to preserve, resulting in higher-fidelity compression that keeps your critical context intact.

The Problem: Context Rot in Long Sessions

Claude Code's 200k-token context window is shared across everything: system prompts, CLAUDE.md files, MCP tool schemas, your messages, and all tool results. As your session grows, quality degrades. Responses slow down. Claude starts "forgetting" earlier decisions. Eventually, auto-compression kicks in at ~83.5% capacity and summarizes your history with no guidance on what to keep.

Each compression cycle is lossy. After 2-3 auto-compressions, your early context is a rough sketch at best.

The Fix: Proactive /compact with Focus Hints

Instead of waiting for auto-compression, run /compact manually at natural breakpoints — and always include a focus hint:

/compact focus on the auth refactor and the new JWT token flow
/compact retain the API contract: POST /api/v2/devices returns { id, status, config }
/compact keep the database migration decisions

The focus hint is free-form text that tells Claude which parts of the conversation to prioritize during compression. Without it, Claude treats everything equally and you lose important details.

When to Compact

  • At subtask boundaries. Finished the backend endpoint? Compact before starting the frontend. Include the API contract in your hint.
  • When responses slow down. This is a signal your context is bloated. Run /cost to confirm, then compact.
  • Before the 80% mark. Aim to compact at ~80% context usage. Auto-compression at 83.5% gives you no say in what's preserved.

/compact vs /clear: Pick the Right Tool

These commands solve different problems:

  • /compact — Use when continuing the same task. Summarizes history (lossy) but keeps you in flow. CLAUDE.md and MCP schemas stay loaded.
  • /clear — Use when switching to an unrelated task. Hard reset. No summary, no retained context. Carrying stale context from task A into task B actively hurts quality.

The Power Move: Short Sessions + Strategic Compaction

The biggest productivity gain comes from structuring work as multiple focused sessions rather than one marathon:

# Session 1: Research and plan → /clear
# Session 2: Implement core logic → /compact focus on the data model
# Session 3: Add tests → /compact focus on the edge cases found
# Session 4: Review and ship

A 20-turn conversation at 100k average context costs ~4x more in tokens than the same work split into 4 focused 5-turn sessions at 25k each. Shorter sessions aren't just higher quality — they're significantly cheaper.

Quick Tip: Verify After Compacting

After running /compact, ask Claude a quick question about a key decision from earlier in the session. If it can't answer accurately, re-state the critical context. Claude doesn't know what it forgot.

Key Takeaway

Always run /compact with a focus hint at natural breakpoints — don't wait for auto-compression to decide what matters for you.
