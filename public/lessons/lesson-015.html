/batch: Parallel Agents for Large-Scale Code Migrations

2026-04-08
batchworkflowproductivity
Quick Quiz
What does /batch do, and how does it prevent parallel agents from stepping on each other's changes?
Click to reveal answer
/batch decomposes a large change into self-contained units and spawns one agent per unit, all running simultaneously. Each agent operates in its own isolated git worktree and branch, so no agent can interfere with another's work. When done, each agent runs tests and creates a PR automatically.

The Problem: Big Refactors Are Sequential Bottlenecks

You need to rename an API field across 40 services, migrate a logging library, or update an auth pattern everywhere it appears. Even with Claude Code, you'd work through each file sequentially — or manually spin up worktree sessions one at a time. That's slow and error-prone.

The Solution: /batch

The /batch skill, announced by Claude Code creator Boris Cherny (@bcherny on X), turns large-scale changes into a parallelized, automated pipeline. You describe the change once, and /batch handles the rest.

How It Works

  1. Research: An orchestrator agent enters plan mode and launches Explore agents to find every file, pattern, and call site affected by your change.
  2. Decompose: The orchestrator splits the work into 5–30 self-contained units — each one small enough for a single agent to handle independently.
  3. Approve: You review the execution plan before any code is touched.
  4. Execute in parallel: One background agent spawns per unit. Each agent gets its own git worktree and branch — full isolation, zero interference.
  5. Test & PR: Each agent runs tests in its worktree, then runs /simplify (a 3-agent code review) before creating a PR.

Using It

Just type /batch followed by a description of what needs to change:

/batch Migrate all services from winston to pino logger. Update imports, configuration, and log level mappings.

Or for a targeted migration:

/batch Replace every use of our deprecated AuthV1 middleware with AuthV2 across all Express route files.

The orchestrator does the thinking — finding all affected files, grouping them into logical units, and asking for your approval before execution begins.

Why Worktree Isolation Matters

The reliability of /batch hinges on git worktree isolation. Without it, parallel agents would create merge conflicts, overwrite each other's changes, or produce broken intermediate states. With worktrees, each agent has a clean working copy. If one agent fails, the others are unaffected.

When to Reach for /batch

  • API migrations: Renaming fields, updating response shapes, or versioning endpoints across many services
  • Library swaps: Replacing one dependency with another (e.g., moment → dayjs, winston → pino)
  • Pattern enforcement: Applying a new coding pattern or removing a deprecated one across the codebase
  • Config updates: Updating environment variables, feature flags, or configuration schemas at scale

If the change touches fewer than 3–4 files, a regular Claude Code session is simpler. /batch shines when the work is repetitive, parallelizable, and spread across many files.

Key Takeaway

Use /batch to turn large, repetitive codebase migrations into parallel agent pipelines — each change isolated in its own worktree, tested, reviewed by /simplify, and shipped as a separate PR.
