The Interview Pattern: Let Claude Ask the Questions Before You Build

2026-05-13
workflow planning best-practices
Quick Quiz
Instead of writing a detailed prompt, how can you get Claude Code to extract requirements from you?
Click to reveal answer
Write a minimal spec and add "interview me in detail using the AskUserQuestionTool." Claude will ask dozens of targeted questions about requirements, edge cases, and trade-offs, then produce a plan far more thorough than most hand-written prompts.

Stop Writing Perfect Prompts

Most people spend 10 minutes crafting a detailed prompt before they let Claude start working. The interview pattern flips this: you write a one-line spec and let Claude extract the details from you through conversation.

The trick is the AskUserQuestionTool — the same tool Claude Code uses in Plan Mode to ask clarifying questions. You can explicitly tell Claude to use it, and it will systematically interview you about every aspect of your feature before writing a single line of code.

How It Works

Start with a minimal description of what you want, then add the interview instruction:

I want to add a real-time chat interface for our support team.

Interview me in detail using the AskUserQuestionTool about technical
implementation, UI/UX, edge cases, security, and trade-offs before
writing any code.

Claude will ask targeted questions one at a time: "Should messages persist after the session ends?" "Do you need typing indicators?" "What's your auth system?" You answer naturally, and Claude builds up a comprehensive picture of what you actually need.

Make It a Reusable Command

Create a custom slash command so you can trigger this pattern on any spec file:

mkdir -p .claude/commands
cat > .claude/commands/interview.md << 'EOF'
---
description: Interview me about the plan
argument-hint: [plan-file]
model: opus
---
Read the plan file $ARGUMENTS and interview me in detail using the
AskUserQuestionTool about technical implementation, UI/UX, concerns,
trade-offs, and anything else relevant. After the interview, produce
a comprehensive implementation plan.
EOF

Now run /interview spec.md on any spec file. One developer reported that a single-line plan document generated 75 clarifying questions and produced a 400-line implementation plan.

The Two-Session Workflow

For best results, split planning and execution into separate sessions:

  1. Session 1 (interview): Run /interview with your spec. Answer all the questions. Claude produces a detailed plan. Save it.
  2. Session 2 (build): Start a fresh session with full context budget. Hand Claude the plan and let it execute.

This keeps the execution session's context window clean — no interview back-and-forth eating into the tokens Claude needs for implementation.

Why This Works Better Than Prompt Engineering

  • You don't know what you don't know. Claude asks about edge cases and trade-offs you wouldn't think to include in a prompt.
  • It's faster. Answering questions is faster than writing a spec from scratch — Claude does the structured thinking for you.
  • It's more thorough. The resulting plan covers auth, error handling, accessibility, and deployment concerns that hand-written prompts usually miss.

Key Takeaway

Don't write the perfect prompt — write a one-line spec and tell Claude to interview you with AskUserQuestionTool. You'll get a better plan in less time.
