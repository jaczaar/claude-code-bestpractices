/goal: Define Done, Then Walk Away

2026-05-13
goalworkflowproductivity
Quick Quiz
How does /goal differ from /loop for keeping Claude working autonomously?
Click to reveal answer
/goal starts the next turn immediately after the previous one finishes and stops when a separate evaluator model confirms your condition is met. /loop starts the next turn on a time interval and stops when you cancel or Claude decides the work is done.

The Problem: Claude Stops When You Don't Want It To

You ask Claude to migrate a module to a new API. It makes progress, then stops and says "I've updated the first three files." You prompt it again. It does three more. You prompt again. This back-and-forth is the #1 friction point for substantial tasks.

/goal eliminates that friction. You define what "done" looks like, and Claude keeps working across as many turns as it takes to get there.

How It Works

Type /goal followed by a completion condition:

/goal all tests in test/auth pass and the lint step is clean

Claude starts working immediately. After each turn, a small fast model (Haiku by default) evaluates whether your condition is met by reading the conversation transcript. If not, Claude gets the evaluator's reason and starts another turn. If yes, the goal clears and control returns to you.

A ◎ /goal active indicator shows elapsed time, turns, and token spend while it runs.

Writing Conditions That Actually Work

The evaluator judges from what Claude has surfaced in the conversation — it doesn't run commands itself. Write conditions around outputs Claude will produce:

  • Measurable end state: /goal npm test exits 0 and no TypeScript errors remain
  • Stated check: /goal every file in src/legacy/ is deleted and the build still passes
  • With constraints: /goal all API endpoints return JSON, no existing tests are modified
  • With a budget: /goal resolve all TODO comments in src/auth or stop after 15 turns

/goal vs. /loop vs. Stop Hooks

All three keep Claude running autonomously, but they trigger differently:

  • /goal — next turn fires immediately after the previous one ends. Stops when the evaluator confirms your condition. Session-scoped.
  • /loop — next turn fires on a time interval. Stops when you cancel or Claude decides it's done. Best for polling tasks.
  • Stop hooks — next turn fires immediately after the previous one ends. Your own script or prompt decides. Lives in settings, applies across sessions.

Combine /goal with auto mode for fully unattended execution: auto mode approves tool calls within each turn, while /goal starts new turns.

Practical Tips

  • Check progress anytime with /goal (no arguments) to see turns, tokens, and the evaluator's latest reason.
  • Cancel early with /goal clear (or stop, off, cancel).
  • Goals survive session resume — if you close and --resume, the goal picks up where it left off.
  • Works in headless mode: claude -p "/goal CHANGELOG.md has an entry for every PR merged this week"

Key Takeaway

Use /goal to define a verifiable completion condition and let Claude work autonomously across as many turns as needed — no babysitting, no re-prompting.
